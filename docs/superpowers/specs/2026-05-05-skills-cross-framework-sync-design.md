# Skills cross-framework sync — design

**Date:** 2026-05-05
**Branch:** `feat/dashboard-docs-wiki` (this is unrelated to the wiki work; new branch will be cut at implementation time)
**Status:** Approved by user

## Problem

Six skills installed via `npx skills add` (5 taste-skill variants + huashu-design) live in `.agents/skills/`. NeuroClaw's `skill-loader.ts` walks `.claude/skills/`, `~/.claude/skills/`, plugin paths, and marketplace paths — **not** `.agents/skills/`. Result: the skills the user just installed are invisible to every NeuroClaw agent.

Beyond fixing that, the user wants any skill (whether installed via `npx skills add`, authored via the dashboard, or installed via `claude plugin install`) to be automatically visible to other agent frameworks the user runs locally — Codex CLI today, with room to add Cursor / Copilot / Pydantic AI later.

## Goals

1. NeuroClaw's local agents see all skills regardless of which install path delivered them.
2. Codex CLI sees the same catalog through native conventions (`AGENTS.md` + per-skill files).
3. Adding a skill in any of the supported install paths automatically updates the Codex export — no manual sync step required for the common cases.
4. The exporter layer is structured so Cursor / Copilot / Pydantic adapters drop in later as one file each.

## Non-goals

- Cursor, Copilot, Pydantic adapters in this iteration. Interface is defined; files added later.
- Filesystem watcher for skill changes.
- Two-way sync. Exports are write-only mirrors of the canonical SKILL.md.
- Translating skill *content* across frameworks. Skills are markdown; the same body works everywhere. "Translate" in this design means structural / format mirroring only.
- Anything from open-design beyond CLI auto-detection (the 31 skills bundle, 72 design systems, question-form discovery, ZIP import are out of scope).

## Architecture

Two roots on disk, one loader, N adapters.

```
Canonical (source of truth)
├── .claude/skills/<name>/SKILL.md        ← authored (editable, source: 'project')
└── .agents/skills/<name>/SKILL.md        ← installed via npx skills add (read-only, source: 'installed')

       │
       ▼  read by
  src/skills/skill-loader.ts  ── listSkills() returns merged catalog
       │
       ├──► NeuroClaw agents  (already wired via buildSkillsBlock — no change)
       │
       ├──► Codex adapter      → AGENTS.md (index) + .codex/skills/<name>.md (per skill)
       │
       └──► [future adapters: Cursor, Copilot, Pydantic — same shape]
```

### Canonical store: two roots, explicit roles

- `.claude/skills/<name>/SKILL.md` — skills authored locally. Editable via dashboard / `createSkill()`. Version-controlled. `source: 'project'`.
- `.agents/skills/<name>/SKILL.md` — skills installed from external sources via `npx skills add`. Tracked in `skills-lock.json`. Read-only — `createSkill`/`updateSkill`/`deleteSkill` refuse to mutate them, matching how `'plugin'` and `'marketplace'` sources are already handled. `source: 'installed'`.

The loader's existing precedence order extends naturally: `project > user > installed > plugin > marketplace`. First writer wins on name collision; an authored skill in `.claude/skills/` shadows an installed one with the same name.

### Exporter interface

```ts
export interface SkillExporter {
  name: string;                                       // 'codex' | 'cursor' | …
  isAvailable(repoRoot: string): Promise<boolean>;    // does this user actually use this CLI?
  export(skills: SkillRecord[], repoRoot: string): Promise<{ written: string[]; skipped: string[] }>;
}
```

`syncSkillExports()` calls `listSkills()` after clearing the loader cache, then runs `export()` for every exporter whose `isAvailable()` returns true. Per-exporter failures are caught and logged; one bad exporter never blocks the rest.

### Codex exporter

- **Format:** index + per-skill files (Question 3 option B).
- `AGENTS.md` at repo root: short index. Each skill gets one entry with name, description, and a relative path to `.codex/skills/<name>.md`. Top of the file carries an auto-generated marker.
- `.codex/skills/<name>.md` per skill: plain mirror of the canonical SKILL.md body, with the frontmatter rendered as a small header at the top of the file.
- **Auto-gen marker** (literal first line of generated `AGENTS.md`):
  ```
  <!-- auto-generated by neuroclaw skills:sync — edits will be overwritten. Add custom Codex instructions to AGENTS.local.md instead. -->
  ```
- **Auto-detection (borrowed from open-design):** `isAvailable()` returns true if `which codex` succeeds OR an `AGENTS.md` already exists at the repo root with the auto-gen marker. If neither, no Codex files are written — repos that don't use Codex stay clean.
- **Don't stomp hand-written `AGENTS.md`:** if `AGENTS.md` exists without the auto-gen marker on its first line, exporter refuses to overwrite and prints: `AGENTS.md exists and was not generated by skills:sync — move it to AGENTS.local.md and re-run`.
- **Custom Codex guidance:** if `AGENTS.local.md` exists, exporter appends its content verbatim to the bottom of the generated `AGENTS.md`.

## Components

**Files that change:**

| File | Change |
|---|---|
| `src/skills/skill-loader.ts` | Add `'installed'` to source union. Add `INSTALLED_ROOT = path.resolve(cwd, '.agents/skills')`. Existing `'project'`-only guards on `createSkill`/`updateSkill`/`deleteSkill` remain correct. |

| `src/dashboard/server.ts` (or wherever startup runs) | Call `syncSkillExports()` once on cold start (fire-and-forget, after schema init). Catches the case where the user ran bare `npx skills add` between server runs. |

## Data flow

```
1. User adds a skill
   ├── Path A: `npx skills add <repo>`            → drops files in .agents/skills/<name>/
   ├── Path B: dashboard "create skill" UI         → calls createSkill() → writes to .claude/skills/<name>/
   └── Path C: `claude plugin install <plugin>`    → drops files via Claude CLI's plugin system

2. Sync trigger fires
   ├── Path A:
   │     • If user ran `npm run skills:add <repo>` → wrapper runs sync after install.
   │     • If user ran bare `npx skills add` → no immediate sync; next NeuroClaw
   │       startup runs syncSkillExports() once. Or user runs `npm run skills:sync`.
   ├── Path B: dashboard endpoint awaits syncSkillExports() inline before responding.
   └── Path C: next NeuroClaw startup picks it up; `npm run skills:sync` (manual)
              works the same.

3. syncSkillExports() runs
   ├── clearSkillCache(); listSkills()              ← fresh catalog
   ├── For each exporter:
   │     if (await exporter.isAvailable(repoRoot))   ← only if user uses that CLI
   │       await exporter.export(skills, repoRoot)
   └── Logs: { exporter: 'codex', written: 7, skipped: 0 }

4. NeuroClaw agents pick up skills automatically
   └── chatStream() already calls buildSkillsBlock() per turn — no change needed.
```

### Triggers

| When | What runs |
|---|---|
| `npm run skills:add <repo>` (recommended wrapper) | `npx skills add` → `npm run skills:sync` |
| Bare `npx skills add` | No immediate sync; next NeuroClaw startup catches it |
| NeuroClaw cold start | `syncSkillExports()` runs once, fire-and-forget |
| Dashboard create / update / delete | route handler awaits `syncSkillExports()` before returning |
| User forces regen | `npm run skills:sync` |

No filesystem watcher. Skills change rarely; the triggers above cover every realistic add path.

## Error handling

| Failure mode | Behavior |
|---|---|
| Skill has malformed frontmatter | `readSkillFile()` already logs a warning and skips. No change. |
| `.agents/skills/` doesn't exist | `walkInstalledSkills()` returns silently (matches `walkSkills()`). |
| Codex exporter fails mid-write | `syncSkillExports()` catches per-exporter, logs, continues. |
| Postinstall hook fires in CI or non-repo context | `syncSkillExports()` no-ops if neither `.claude/skills/` nor `.agents/skills/` exists. |
| Dashboard sync fails | Mutation still succeeds; sync error returned in response body as a warning. Agent registry is the source of truth, exports are downstream. |
| `AGENTS.md` exists without auto-gen marker | Refuse to overwrite; print remediation message. |

## Testing

The project has no test suite. Correctness gate:

1. **Type check:** `npx tsc --noEmit` after every change.
2. **Manual verification:**
   - Run `npm run skills:sync` → assert `AGENTS.md` and `.codex/skills/*.md` appear with all 13 skills (7 project + 6 installed).
   - Start `npm run dev`, send a message through Alfred, confirm a taste-skill or huashu-design body now appears in the system prompt (loader sees `.agents/skills/`).
   - Open dashboard, create a test skill via the UI, confirm `AGENTS.md` regenerates and includes it.
   - Delete the test skill, confirm it disappears from `AGENTS.md`.
   - Move `AGENTS.md` aside, create one with no marker, run sync, confirm the refusal message.

## Decisions log

- **Q1: Which agent surfaces?** F — NeuroClaw local agents + Codex now; pluggable adapters for the rest.
- **Q2: Canonical store?** C — both `.claude/skills/` (authored) and `.agents/skills/` (installed), loader walks both.
- **Q3: Codex export shape?** B — index `AGENTS.md` + per-skill files at `.codex/skills/<name>.md`.
- **From open-design:** borrowed CLI auto-detection for the `isAvailable()` check. Everything else from open-design is out of scope.
